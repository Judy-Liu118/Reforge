## Coverage boundary (not a metric)

- D1″ probes **only the timeout sub-path** of deliberate STOP (`failure_mode == "timeout"`, set in `classifier.py:36-40` when sandbox watchdog kills the process). The **terminal_intentional sub-path** (`is_expected_failure=True AND retryable=False`, set when IntentStage classifies the user request as EXPECTED_ERROR / TRACEBACK_DEMO) is NOT exercised here — constructing a prompt that triggers it would leak intent into the calibration corpus. See `docs/KNOWN_LIMITATIONS.md` L3 for the architectural gap and why a pattern-based detector is deferred.
- BIRD result-direction comparison (governor vs naive solve / recovery rates) is **out of Phase 0 scope** and is intentionally not computed in this report. See Phase 1 (`docs/eval/...` to be written) for the headline ablation.

## Observed runtime behavior (descriptive, not interpretive)

The calibration surfaced two runtime/corpus behaviors that the
pre-registered design did not anticipate. They are recorded here as
**facts about the instrument and the corpus**; their thesis-level
interpretation lives in `docs/eval/PHASE0_METRICS.md` v4 and
`docs/KNOWN_LIMITATIONS.md` L5 / L6. No governor-vs-naive direction
claim is drawn here — that remains Phase 1 territory.

### O1. D1″ under naive mode — codegen LLM adapted away from long sleep

Pre-registered prediction (`PHASE0_CORPUS.md` D1″): naive runs
`initial + 3 retries = 4 attempts × ~30 s ≈ 120 s wall`. Each attempt
re-writes `time.sleep(120)`, hits the watchdog, naive RETRYs, until
`retry_count == max_retry` → budget-exhausted STOP.

Observed (3/3 naive seeds on D1″):

| seed | attempts | action | retry_count | policy_reason |
|---|---|---|---|---|
| 0 | 3 | ACCEPT | 2 | `naive: exit_code == 0` |
| 1 | 3 | ACCEPT | 2 | `naive: exit_code == 0` |
| 2 | 2 | ACCEPT | 1 | `naive: exit_code == 0` |

The codegen LLM, after seeing its previous attempt time out, did not
re-emit `time.sleep(120)` on the next attempt. Instead it shortened
the sleep, removed it, or rewrote the script to exit cleanly. The
naive baseline then saw `exit_code == 0` and ACCEPTed — a codegen
behavioral adaptation under retry pressure, not a runtime decision.

The governor leg of D1″ still exhibited the deliberate-STOP timeout
path correctly (3/3 seeds, `action == "STOP"`, `failure_mode ==
"timeout"`, `policy_reason == "timeout"`, `retry_count == 0`). The
gate the calibration probes — that the timeout branch of
`RetryPolicy.decide` is reachable — is GO. What this observation
rebuts is **only** the pre-registered prediction about naive's
wall-clock cost on timeout decoys, not the deliberate-STOP code path
itself.

Phase 2 methodological consequence recorded in
`docs/eval/PHASE0_METRICS.md` v4 (timeout-class deliberate-STOP
efficiency headline deferred); architectural note recorded in
`docs/KNOWN_LIMITATIONS.md` L5.

### O2. bird_1313 governor — internal LLM evaluator rejected correct answers

Observed (3/3 governor seeds on `bird_1313_student_club`):

| seed | attempts | action | policy_reason | runtime_outcome | passed (comparator) | recovered |
|---|---|---|---|---|---|---|
| 0 | 4 | STOP | `evaluation_failed` | FAILED | True | True |
| 1 | 4 | STOP | `evaluation_failed` | FAILED | True | True |
| 2 | 4 | STOP | `evaluation_failed` | FAILED | True | True |

The runtime's internal LLM evaluator (set in
`state.semantic_state.evaluation_result`) returned `passed=False` for
the attempt that produced the SQL-comparator-correct row. The
runtime then RETRY'd until `retry_count == max_retry` and emitted a
`retry_limit_reached_on_eval_fail` STOP, recording the case as
`runtime_outcome == "FAILED"` even though the answer was right by
ground truth.

Calibration consequence:

- The `bird_recoverability` gate fires correctly because the
  calibration grades BIRD via the SQL comparator
  (`reforge.runtime.sql.comparator`), not `runtime_outcome` — same as
  `SqlBenchSession._grade`. Three governor recoveries on bird_1313
  satisfy the gate at ≥ 1.
- Phase 1 BIRD measurement MUST use the SQL comparator throughout.
  `runtime_outcome=FAILED` and `policy_reason=evaluation_failed` are
  **misleading signals** when the LLM evaluator returns
  false-negatives on correct outputs.
- The governor's recovery rate carries an implicit upper bound set
  by the LLM evaluator's precision: an evaluator false-negative on a
  correct output makes the runtime burn additional retries on an
  already-solved case. Recorded in `docs/KNOWN_LIMITATIONS.md` L6
  with the Phase 1 sensitivity-appendix requirement to quantify the
  paired-delta impact.
