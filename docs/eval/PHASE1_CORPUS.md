## Design (inherited, restated for one-page reading)

- **Axis 1 only**: governor (`REFORGE_GOVERNOR_BYPASS` unset) vs naive
  (`=1`). No memory axis, no interaction effects (out of scope per v4).
- **N_seeds = 5** (headline-axis budget; df=4, t=2.776).
- **Report unit**: per-seed paired delta, then Student-t 95% CI over
  seeds (`MultiSeedDriver.summarise()` machinery).
- **passed/failed field-of-record**: `reforge.runtime.sql.comparator`
  on the gold-SQL result rows. `runtime_outcome`, `policy_reason`, and
  the internal `evaluation_result.passed` are banned as pass signals
  (KNOWN_LIMITATIONS L6).
- **Headline rule**: a delta whose 95% CI crosses zero is "no
  significant effect (CI includes 0)" — tables only, never headline copy.

## Corpus selection rule (a priori, deterministic)

Pool: BIRD dev (`data/bird/dev.json`, 1534 questions, dev.zip SHA256
pinned in `PHASE0_CORPUS.md`). Exclusions, applied in order:

| Filter | Rule | Removed |
|---|---|---|
| SQLite-dialect functions | gold SQL matches `\b(JULIANDAY\|STRFTIME\|DATETIME\|IIF)\s*\(` (case-insensitive) | 149 |
| Phase-0 calibration picks | `question_id ∈ {7, 1313, 354, 697, 838}` | 5 |

Remaining pool: **1380** — 868 simple / 395 moderate / 117 challenging.

**Stratified sample, N = 20** — 12 simple / 6 moderate / 2 challenging
(≈ the pool's 63/29/8 split). Procedure: per stratum, sort candidates
by `question_id` ascending, then `random.Random(20260710).sample(...)`
with a single RNG instance consumed in the order simple → moderate →
challenging. Reproduced by `select_phase1_case_ids()` in
`reforge/benchmark/phase1/corpus.py`; the frozen list below is the
record, the code asserts against it.

Rationale notes:

- **Dialect filter** keeps grading about recovery behavior, not SQLite
  trivia; it also matches the Phase-0 pool audit so the two phases
  describe the same population.
- **Calibration exclusion is at question level, not db level.** The M1
  memory-loop repair and the driver-isolation fix were motivated by
  observations on those five specific cases; their *databases* were
  never tuned against, so sibling questions on the same dbs stay
  eligible.
- **No evidence filter.** Phase 0 dropped evidence-heavy questions when
  hand-picking recoverable-JOIN probes — a probe-design choice, not a
  validity rule. Phase 1 keeps BIRD's evidence field as-is in the
  prompt; it is identical across arms, so it cancels in the paired
  delta.

## Run protocol

- **Total runs**: 20 cases × 2 modes × 5 seeds = **200**.
- **Loop order**: mode outer → seed → the 20 cases in frozen-list
  order (identical order in both arms, so within-leg memory-accrual
  ordering effects are arm-invariant and cancel in the paired delta).
- **Memory hygiene**: per-(mode, seed) leg cold-start — the leg scopes
  `REFORGE_PROJECT_DIR` + `REFORGE_HOME` to a fresh tmp dir (same
  mechanism the repaired Phase-0 driver uses). Within a leg, memory
  accrues across the 20 cases; that within-session accrual is part of
  the system under test, not contamination.
- **Token accounting**: harness-side
  `token_accounting(case_id, seed)`; `usage=None → -1` sentinel and the
  >20%-excluded no-headline rule per v4.
- **Durability**: the driver appends one JSON line per completed run to
  `docs/eval/phase1_records.jsonl` and can resume. **Resume granularity
  is the full (mode, seed) leg**: a leg with fewer than 20 records is
  discarded and re-run, because a mid-leg restart would silently reset
  the within-leg memory accrual that completed legs carry.

## Sensitivity appendix operationalization (v4 §4)

v4 requires quantifying the internal LLM evaluator's false-negative
rate and its effect on the headline. Locked operationalization:

- The driver consumes `RuntimeRunner.stream()` and, per attempt, pairs
  the `execution` node's `exec_state.stdout` with the following
  `evaluation` node's `evaluation_result` (score, passed). This is
  harness-side observation only — no runtime change.
- **Attempt-level false negative** := the SQL comparator confirms the
  attempt's stdout matches gold rows AND the internal evaluator
  rejected it (`passed=False`).
- Report per (mode, seed): FN attempts / all attempts, and FN attempts
  / comparator-correct attempts.
- **Asymmetry decision rule**: compute the per-seed paired delta of the
  attempt-level FN rate (governor − naive). If its 95% CI excludes
  zero, evaluator pressure is asymmetric and every headline claim
  carries an explicit caveat naming it. If the CI includes zero, the
  paired subtraction is taken to cancel the common noise (v4's
  "robust by construction" case) and the headline stands unqualified.
- Run-level view (KNOWN_LIMITATIONS L6 shape) is also tabulated:
  runs with comparator `passed=True` but `runtime_outcome == "FAILED"`.
